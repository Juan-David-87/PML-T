Data Engineering

Complete preprocessing pipeline for the Digital Government Index — Historical dataset. Each step is documented with its objective, rationale, and applied operations.

Step 1

Data Cleaning

Standardise the raw dataset by fixing column names, removing duplicates, correcting data types, and enforcing domain rules so that downstream steps operate on structurally sound data.

What

Detect and correct inaccurate, inconsistent, or structurally corrupt records before any analysis.

Why

Dirty data propagates errors throughout the pipeline. Inconsistent names and wrong types bias every metric.

How

Rename columns to snake_case, drop duplicates, strip whitespace, uppercase text, coerce types, and remove out-of-range records.

  • ✓ Column name standardisation — renamed all 18 columns to lowercase ASCII snake_case, removing Spanish accents and spaces.
  • ✓ Exact duplicate removal — dropped rows where all 18 fields are identical.
  • ✓ Whitespace trimming — applied str.strip() to all text columns.
  • ✓ Text case normalisation — all categorical columns converted to uppercase.
  • ✓ Data type enforcement — year/ID columns cast to Int64; score columns kept as float64.
  • ✓ Domain rule validation — removed records where scores fall outside [0, 100] or quintiles outside [1, 5].
Step 2

Null Value Handling

Systematically identify and treat missing data — dropping columns with excessive absence, and imputing reasonable values based on each column's statistical distribution.

What

Systematic identification and treatment of missing data.

Why

Most ML algorithms cannot handle NaN values. Missing scores may represent entities that did not participate in a given measurement cycle.

Strategy

Numeric scores → median. Quintil → mode. Categoricals → 'UNKNOWN'. Columns with >50 % nulls are dropped.

  • ✓ High-null column removal (>50 %) — any column with more than half the records missing is dropped.
  • ✓ Median imputation — applied to puntaje_entidad, promedio_grupo_par, maximo_grupo_par, minimo_grupo_par, percentil_grupo_par.
  • ✓ Mode imputation — applied to quintil_grupo_par (ordinal integer 1–5).
  • ✓ 'UNKNOWN' label — missing values in orden, sector, naturaleza_juridica, departamento, municipio, indice are filled with an explicit label to preserve missingness as a learnable signal.
Step 3

Data Transformation

Derive new variables that expose latent structure — such as how far an entity is from the best performer in its group, or whether it exceeds the group average.

What

Create new columns by applying mathematical or logical operations to existing ones.

Why

Raw score columns alone cannot encode contextual relationships like gap to the best peer or relative group position.

Result

6 new numeric features and 1 categorical tier variable added to the dataset.

  • ✓ gap_to_max = maximo_grupo_par − puntaje_entidad — distance from the top performer in the peer group.
  • ✓ gap_to_avg = puntaje_entidad − promedio_grupo_par — positive means above average, negative means below.
  • ✓ spread_grupo_par = maximo − minimo — how heterogeneous the peer group is.
  • ✓ relative_performance = (score − min) / (max − min) — normalises each entity to [0, 1] within its peer group.
  • ✓ performance_tier — discretised into four bands: HIGH (≥80), MEDIUM-HIGH (60–79), MEDIUM-LOW (40–59), LOW (<40).
  • ✓ year_offset = vigencia − min(vigencia) — anchors the year at zero for trend modelling.
Step 4

Categorical Variable Encoding

Convert text categories into numeric representations so that ML algorithms can process them via arithmetic or distance metrics.

What

Translation of categorical text variables into numeric form.

Why

Most ML models require numeric inputs. Without encoding, columns like sector are ignored or cause errors.

Scheme

Label Encoding for ordinal columns, One-Hot (top-N) for nominals, Frequency Encoding for high-cardinality columns.

  • ✓ Label Encoding → orden, naturaleza_juridica — few distinct values, each category mapped to an integer.
  • ✓ One-Hot Encoding (top 10) → sector — top 10 categories encoded; remaining grouped as 'OTHER' to limit dimensionality.
  • ✓ Frequency Encoding → indice, departamento — high-cardinality columns replaced with their relative frequency, avoiding hundreds of new columns.
  • ✓ Ordinal Encoding → performance_tier — LOW=0, MEDIUM-LOW=1, MEDIUM-HIGH=2, HIGH=3, preserving the inherent order.
Step 5

Feature Selection

Retain only the most informative features from the full candidate pool to reduce dimensionality, mitigate overfitting, and improve model interpretability.

What

Select a subset of features from the full pool to reduce dimensionality and noise.

Why

After encoding and transformation the dataset has 30+ columns. Irrelevant or redundant features increase training time and can hurt generalisation.

Methods

Variance Threshold → Correlation Filter → SelectKBest / F-regression, applied in sequence.

  • ✓ Variance Threshold (0.01) — features whose variance is below 0.01 are nearly constant and carry no information; dropped automatically.
  • ✓ Correlation Filter (|r| > 0.95) — for each highly correlated pair, the second column is dropped to remove multicollinearity.
  • ✓ SelectKBest · F-regression (k = 10) — ranks remaining features by their univariate F-statistic against percentil_grupo_par; the top 10 are retained.
Step 6

Feature Engineering

Build domain-informed temporal features that capture performance trajectories — information invisible in any single year's snapshot.

What

Construct new features using domain knowledge to leverage the multi-year structure of the dataset.

Why

A single year's score cannot show whether an entity is improving, stagnating, or declining. Temporal features expose these trends.

Result

6 new temporal and contextual features computed via groupby, diff, and rolling operations.

  • ✓ yoy_change — year-over-year score delta per entity-index pair; positive = improvement, negative = regression.
  • ✓ rolling_2y_avg — 2-year rolling mean that smooths short-term volatility and reflects a stable performance baseline.
  • ✓ improving — binary flag (1 = score increased vs prior year); simplifies the trend signal for classification tasks.
  • ✓ entity_historical_avg — entity-level average across all years; provides a baseline independent of year or index.
  • ✓ deviation_from_entity_avg — current score minus entity's own historical average; identifies exceptionally good or poor years.
  • ✓ urgency_index = gap_to_max × year_offset — amplifies urgency for entities that have been far from the group maximum for many years.
Step 7

Normalisation & Scaling

Bring all numeric features to comparable scales so that no single column dominates distance-based or gradient-based algorithms due to its absolute magnitude.

What

Transform numeric features so they occupy a common range or distribution.

Why

Without scaling, a feature ranging 0–100 and one ranging 0–1 are treated very differently by algorithms like SVM or KNN, even if equally important.

Three scalers

MinMaxScaler for bounded scores, StandardScaler for deltas, RobustScaler for ordinal variables prone to outliers.

  • ✓ MinMaxScaler → suffix _minmax — applied to score and group-benchmark columns (puntaje_entidad, gap_to_max, gap_to_avg, spread_grupo_par, relative_performance, etc.).
  • ✓ StandardScaler (Z-score) → suffix _zscore — applied to temporal and deviation columns (percentil_grupo_par, yoy_change, rolling_2y_avg, deviation_from_entity_avg, urgency_index).
  • ✓ RobustScaler → suffix _robust — applied to year_offset and quintil_grupo_par; uses IQR instead of mean/std, making it resilient to outliers in ordinal distributions.
